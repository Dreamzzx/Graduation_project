### 2.5 线程安全队列 (VideoFrameQueue)

**核心功能：**
- 生产者-消费者模式
- 线程安全
- 支持超时等待
- 支持丢弃旧帧

**关键实现：**
```cpp
template <typename T>
class VideoFrameQueue {
private:
    std::queue<T> queue;
    std::mutex mtx;
    std::condition_variable cv;
    size_t max_queue_size = 1;  // 队列大小=1，最低延迟
    
public:
    // 推入帧，队列满时丢弃最旧的
    void Push(T&& frame) {
        std::lock_guard<std::mutex> lock(mtx);
        if (queue.size() >= max_queue_size) {
            queue.pop();  // 丢弃旧帧
        }
        queue.push(std::move(frame));
        cv.notify_all();
    }
    
    // 获取最新帧，丢弃中间帧
    std::optional<T> PopLatestNonBlocking(bool& is_running) {
        std::unique_lock<std::mutex> lock(mtx);
        if (queue.empty()) return std::nullopt;
        
        T data = std::move(queue.back());  // 取最新
        while (!queue.empty()) queue.pop(); // 清空队列
        return data;
    }
};
```

## 三、关键技术点

### 3.1 GPU加速全链路

```
摄像头(MJPG) → CUDA解码 → GPU显存 → YOLO推理 → NVENC编码 → RTSP推流
     ↓            ↓          ↓          ↓          ↓          ↓
  压缩数据     硬件解码    零拷贝    GPU计算    硬件编码    网络传输
```

**为什么快？**
1. **CUDA解码**: MJPG块级并行解码
2. **零拷贝**: 数据始终在GPU显存，无CPU-GPU传输
3. **NVENC编码**: 专用硬件电路，不占用GPU计算资源
4. **无B帧**: 消除帧间依赖，降低延迟

### 3.2 时间戳同步 (90kHz)

**为什么是90kHz？**
```
90kHz = 27MHz / 300  (MPEG标准)

优点：能整除所有常见帧率
- 24fps:  90000/24 = 3750 (整数)
- 25fps:  90000/25 = 3600 (整数)
- 30fps:  90000/30 = 3000 (整数)
- 29.97fps: 90000/29.97 ≈ 3003 (整数)

如果用1MHz:
- 30fps: 1000000/30 = 33333.33 (小数，会累积误差)
```

**PTS计算：**
```cpp
// 方法1: 帧序号计算
pts = frame_index * (90000 / fps);

// 方法2: 实际时间计算 (更准确)
auto now_ms = duration_cast<milliseconds>(now.time_since_epoch()).count();
pts = (now_ms - start_time_ms) * 90;  // 1ms = 90 ticks
```

### 3.3 H.264编码参数

**NAL单元类型：**
| NAL Type | 名称 | 说明 |
|----------|------|------|
| 1 | 非IDR帧 | P帧 |
| 5 | IDR帧 | I帧，可独立解码 |
| 7 | SPS | 序列参数集 |
| 8 | PPS | 图像参数集 |

**帧类型：**
```
GOP结构: I P P P P P P P P P P P I P P ...
         ├───── GOP ─────┤├───── GOP ─────┤
         
I帧: 关键帧，可独立解码，体积大
P帧: 预测帧，依赖前面的帧，体积小
B帧: 双向预测帧，依赖前后帧，延迟高 (本项目禁用)
```

### 3.4 延迟优化

**延迟组成：**
| 环节 | 延迟 | 优化方法 |
|------|------|----------|
| 视频采集 | 10-20ms | 降低分辨率/帧率 |
| CUDA解码 | 5-10ms | 使用硬件解码 |
| YOLO推理 | 150-300ms | 使用GPU、量化模型 |
| NVENC编码 | 10-20ms | p1+ull预设 |
| 网络传输 | 50-100ms | 使用TCP、大buffer |
| 服务器缓冲 | 100-200ms | 调整服务器配置 |

**优化配置：**

## 四、面试常见问题

### 4.1 基础问题

**Q1: 项目的主要功能是什么？**
> 这是一个智能视频推流系统，核心功能包括：实时视频采集、YOLOv11人物检测、RTSP推流、智能录像。技术特点是GPU全链路加速，端到端延迟200-400ms。

**Q2: 为什么选择RTSP而不是RTMP？**
> RTSP基于TCP或UDP，支持低延迟场景；RTMP基于TCP，延迟通常2-5秒。对于实时监控场景，RTSP更合适。RTMP更适合直播场景。

**Q3: 为什么用ONNX Runtime而不是PyTorch？**
> ONNX Runtime是推理引擎，比PyTorch快2-5倍；模型格式标准化，跨平台部署方便；支持GPU加速，显存占用低。

### 4.2 技术深度问题

**Q4: NVENC为什么比libx264快？**
```
1. 硬件vs软件: NVENC是专用编码电路，libx264是CPU软件实现
2. 并行度: NVENC有数千个编码单元并行工作
3. 无B帧: 配置max_b_frames=0，消除帧间依赖
4. 零拷贝: GPU显存直接编码，无CPU介入
```

**Q5: CUDA解码什么时候反而更慢？**
```
1. 小分辨率: CUDA启动开销(~1ms)占比过大
2. GPU被占用: YOLO推理占用GPU，解码排队等待
3. MJPG格式: 解码本身很轻量，CPU足够
```

**Q6: 队列大小为什么设为1？**
```
目的: 最低延迟
原理: 队列越大，缓冲帧越多，延迟越高
问题: 处理不及时会丢帧，无法应对突发情况
改进: 动态调整队列大小，或使用双缓冲
```

**Q7: SPS/PPS是什么？为什么要注入？**
```
SPS: 序列参数集，包含分辨率、帧率、profile等
PPS: 图像参数集，包含量化参数、参考帧列表等

注入原因: RTSP协议要求每个I帧前必须有SPS/PPS
不注入后果: 播放器无法初始化解码器，黑屏/花屏
```

**Q8: 90kHz时间基怎么来的？**
```
来源: MPEG标准，90kHz = 27MHz/300
优点: 能整除所有常见帧率(24/25/30/60fps)
好处: PTS增量是整数，无累积误差
```

### 4.3 扩展问题

**Q9: 如何支持10路视频？**
```
方案1: 多实例并行 - 10个采集+检测+推流实例
方案2: Batch推理 - 10帧打包一次推理，显存从20GB降到4-6GB
方案3: 共享检测器 - 10路轮流使用一个检测器，帧率下降

推荐: Batch推理 + GPU显存监控 + 动态降级
```

**Q10: 公网推流延迟会变成多少？**
```
局域网: 200-400ms
公网预估: 500-1500ms

增加原因:
- 网络传输: +100-500ms
- 服务器转发: +50-200ms
- 客户端缓冲: +100-300ms

优化方案:
- 降低分辨率/码率
- 使用WebRTC替代RTSP
- CDN加速、边缘节点部署
```

**Q11: 检测数据如何存储？如何做行为分析？**
```
存储方案:
- 结构化数据: SQLite/MySQL (帧ID、时间戳、边界框)
- 截图: 对象存储/本地文件

行为分析:
- 目标追踪: DeepSORT/ByteTrack，分配唯一ID
- 轨迹分析: 徘徊检测(停留时间)、奔跑检测(速度)
- 异常检测: 跌倒(宽高比突变)、打架(多人交互)
```
